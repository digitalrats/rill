# План миграции rill-patchbay на архитектуру очередей rill-core

## 1. Обзор

Цель миграции — привести `rill-patchbay` в соответствие с архитектурой очередей `rill-core`. Текущая реализация использует собственную очередь `RtQueue<ParameterCommand>` и строковые идентификаторы параметров. После миграции будут использоваться стандартные очереди `rill-core`: `CommandQueue`, `TelemetryQueue`, типы команд (`SetParameter`, `AutomatonCommand`), идентификаторы (`ParameterId`, `PortId`, `NodeId`) и интеграция с `SignalSource`.

## 2. Анализ текущей архитектуры

### 2.1 Зависимости
- `rill-patchbay` зависит от `rill-core` версии 0.3.0 (локальный путь).
- Используется `crossbeam-channel`, `parking_lot`, `rand`, `log`.
- Внутренняя очередь `RtQueue<ParameterCommand>` импортируется из `rill_core::queue::RtQueue` (определение отсутствует в исходниках, возможно, устаревший тип).

### 2.2 Ключевые модули
- **`manager.rs`** – центральный координатор, содержит `RtQueue<ParameterCommand>`.
- **`control.rs`** – маппинг событий, также использует `RtQueue`.
- **`parameter_auto.rs`** – автоматизированные параметры (не связаны с очередями).
- **`automaton/`** – реализация автоматов (LFO, огибающие, секвенсоры).
- **`lib.rs`** – реэкспортирует `RtQueue`.

### 2.3 Типы данных
- `ParameterCommand` – содержит `node_id: NodeId`, `param: String`, `value: f32`, `timestamp: u64`.
- `PatchbayEvent` – события для логирования.
- `Automaton`, `BoxedAutomaton` – базовые трейты автоматов.

### 2.4 Публичный API
- `PatchbayManager::new(config, command_queue)` – принимает `Arc<RtQueue<ParameterCommand>>`.
- `PatchbayControl::new(command_queue)` – аналогично.
- Методы добавления автоматов (`add_lfo`, `add_envelope`, …).
- Отправка команд через `command_queue.send(ParameterCommand {…})`.

## 3. Целевая архитектура

### 3.1 Очереди rill-core
- **`CommandQueue<T>`** – типобезопасная MPMC очередь, поддерживает любые типы, реализующие трейт `Command`.
- **`TelemetryQueue`** – специализированная очередь для телеметрии (логирование, мониторинг).
- **Сигналы** – источник команды (`SignalSource`) указывается в `SetParameter`.

### 3.2 Типы команд
- `SetParameter` – содержит `port: PortId`, `parameter: ParameterId`, `value: f32`, `source: SignalSource`, `timestamp: u64`.
- `AutomatonCommand` – управление автоматами (включение/выключение, изменение параметров, соединение).
- `CommandEnum` – объединение всех типов команд, может передаваться через `CommandQueue<CommandEnum>`.

### 3.3 Идентификаторы
- `ParameterId` – типобезопасный идентификатор параметра (вместо строк).
- `PortId` – идентификатор порта узла.
- `NodeId` – идентификатор узла.

### 3.4 Интеграция с audio graph
Команды `SetParameter` предназначены для изменения параметров узлов в `rill-graph`. Источник (`SignalSource`) указывает, откуда пришла команда (автомат, сенсор, серво, ручное управление).

## 4. Этапы миграции

### Этап 0: Подготовка (неделя 1)
- Создать ветку `feature/patchbay-queue-migration`.
- Обновить `Cargo.toml` rill-patchbay: убедиться, что используется актуальная версия `rill-core` (0.3.0).
- Добавить зависимости для тестирования (при необходимости).

### Этап 1: Замена RtQueue на CommandQueue (неделя 2)
- Удалить импорт `rill_core::queue::RtQueue`.
- Вместо него импортировать `rill_core::queues::{CommandQueue, CommandEnum, SetParameter, SignalSource}`.
- Изменить тип `command_queue` с `Arc<RtQueue<ParameterCommand>>` на `Arc<CommandQueue<CommandEnum>>`.
- Обновить конструкторы `PatchbayManager::new` и `PatchbayControl::new` соответственно.
- Временно сохранить преобразование `ParameterCommand` → `SetParameter` внутри методов отправки.

### Этап 2: Переход с ParameterCommand на SetParameter (неделя 2–3)
- Удалить структуру `ParameterCommand` из `manager.rs`.
- Заменить все использования `ParameterCommand` на `SetParameter`.
- Преобразовать строковый `param: String` в `ParameterId` с помощью `ParameterId::new(&param).unwrap()` (или использовать существующие идентификаторы).
- Добавить поле `source: SignalSource` (например, `SignalSource::Automaton(id)`).
- Обновить методы создания команд (`new_parameter_command` и т.п.).

### Этап 3: Интеграция TelemetryQueue (неделя 3)
- Добавить поле `telemetry_queue: Option<Arc<TelemetryQueue>>` в `PatchbayManager` и `PatchbayControl`.
- При отправке команд дублировать важные события в `TelemetryQueue` (например, `Telemetry::ParameterValue`).
- Опционально: добавить конфигурацию для включения/выключения телеметрии.

### Этап 4: Интеграция SignalQueue для автоматов (неделя 4)
- Определить, требуется ли отдельная очередь для сигналов между автоматами. Возможно, использовать `CommandQueue<AutomatonCommand>`.
- Добавить подписку автоматов на команды (например, `AutomatonCommand::SetParameter` для изменения параметров автомата).
- Реализовать в `AutomatonManager` обработку входящих команд из очереди.

### Этап 5: Рефакторинг AutomatonManager (неделя 5)
- Переработать `AutomatonManager` для работы с очередями команд.
- Автоматы должны генерировать команды `SetParameter` и отправлять их в `CommandQueue`.
- Убрать прямые вызовы изменения параметров, заменить на отправку команд.
- Обеспечить возможность обработки команд `AutomatonCommand` для управления самими автоматами.

### Этап 6: Обновление публичного API (неделя 6)
- Изменить сигнатуры публичных методов, чтобы они принимали `ParameterId` вместо `String`.
- Обновить документацию и примеры.
- Рассмотреть обратную совместимость: возможно, оставить deprecated-методы со строковыми параметрами, которые конвертируют в `ParameterId`.

### Этап 7: Тестирование и отладка (неделя 7)
- Запустить существующие тесты, исправить ошибки компиляции.
- Написать интеграционные тесты для проверки работы очередей.
- Проверить совместную работу с `rill-graph` (если есть возможность).
- Профилирование производительности.

### Этап 8: Финализация (неделя 8)
- Удалить устаревший код (например, `ParameterCommand`, `RtQueue`).
- Провести ревью кода.
- Слить ветку в основную ветку разработки.

## 5. Конкретные изменения в файлах

### `manager.rs`
- Заменить поле `command_queue: Arc<RtQueue<ParameterCommand>>` на `command_queue: Arc<CommandQueue<CommandEnum>>`.
- Добавить поле `telemetry_queue: Option<Arc<TelemetryQueue>>`.
- Обновить метод `new` и `with_command_queue`.
- В методе `send_command` преобразовать `ParameterCommand` в `SetParameter` и обернуть в `CommandEnum::SetParameter`.
- Добавить отправку телеметрии при необходимости.

### `control.rs`
- Аналогичные изменения для `PatchbayControl`.
- Маппинг событий должен создавать `SetParameter` с правильным `SignalSource`.

### `parameter_auto.rs`
- Не требует изменений, так как не использует очереди.

### `automaton/mod.rs`
- Расширить трейт `Automaton` возможностью генерировать команды (метод `generate_commands`).
- Добавить обработку входящих команд `AutomatonCommand` в `AutomatonManager`.
- Возможно, добавить поле `command_queue` в `AutomatonManager`.

### `lib.rs`
- Удалить реэкспорт `RtQueue`.
- Реэкспортировать `CommandQueue`, `CommandEnum`, `SetParameter`, `SignalSource`, `TelemetryQueue`.
- Обновить прелюдию.

### `Cargo.toml`
- Убедиться, что версия `rill-core` актуальна (0.3.0). При необходимости обновить до последней.

## 6. Публичный API и обратная совместимость

### Критические изменения
- Тип очереди изменён с `RtQueue<ParameterCommand>` на `CommandQueue<CommandEnum>` – **breaking change**.
- Идентификаторы параметров теперь `ParameterId` вместо `String` – **breaking change**.
- Конструкторы `PatchbayManager::new` и `PatchbayControl::new` требуют `CommandQueue<CommandEnum>`.

### Стратегия обратной совместимости
1. **Методы-обёртки**: предоставить deprecated-функции, которые конвертируют старые типы в новые.
   - `PatchbayManager::with_rt_queue(queue: Arc<RtQueue<ParameterCommand>>)` – преобразует очередь и создаёт новую.
   - `add_automaton_with_string_id` – принимает строки, конвертирует в `ParameterId`.
2. **Фаза устаревания**: объявить устаревшие функции с атрибутом `#[deprecated]` на один релиз.
3. **Документация**: обновить примеры и документацию, указав на новые типы.

Если обратная совместимость невозможна, версию `rill-patchbay` увеличить до `0.4.0` (согласно SemVer).

## 8. Риски и способы их минимизации

### Риск 1: Потеря производительности из-за накладных расходов CommandQueue
- **Митигация**: провести профилирование, убедиться, что `CommandQueue` на базе `crossbeam-channel` не уступает `RtQueue`. При необходимости оптимизировать (например, использовать bounded очередь с политикой `DropOldest`).

### Риск 2: Ошибки преобразования идентификаторов (ParameterId::new может вернуть ошибку)
- **Митигация**: добавить валидацию строк при создании `ParameterId`. Использовать `unwrap` только в доверенном коде; в публичном API возвращать `Result`.

### Риск 3: Нарушение обратной совместимости для существующих пользователей
- **Митигация**: предоставить deprecated-функции и четко документить изменения. Выпустить миграционный гайд.

### Риск 4: Сложность отладки из-за использования нескольких очередей
- **Митигация**: добавить подробное логирование (через `log` crate) и использовать `TelemetryQueue` для мониторинга.

### Риск 5: Неполная интеграция с SignalQueue (отсутствие реализации в rill-core)
- **Митигация**: если `SignalQueue` ещё не реализован, временно использовать `CommandQueue` с типом `AutomatonCommand`. Согласовать с командой разработки `rill-core` план добавления `SignalQueue`.

## 9. Заключение

Предложенный план разбивает миграцию на восемь этапов, каждый из которых можно выполнять и тестировать независимо. Основные breaking changes касаются типа очереди и идентификаторов параметров. Рекомендуется увеличить мажорную версию `rill-patchbay` до `0.4.0` после завершения миграции.

Следующий шаг: утверждение плана, после чего можно приступать к реализации.
